5.3 Dua Test Case Paling Penting

1. TC-07: isNearDeadline() dengan format tanggal invalid

Alasan: Deadline diinput sebagai string dari user/database. Jika format tidak valid (misal: "bukan-tanggal"), fungsi harus return false, bukan crash. Test ini memastikan error handling bekerja — tanpa ini, aplikasi bisa crash setiap kali ada data korup di database.

2. IT-01: Konversi Task API→View→API mempertahankan data

Alasan: Aplikasi ini memiliki 2 model berbeda untuk Task (API model dan View model). Data harus dikonversi bolak-balik tanpa kehilangan informasi. Test ini memastikan bahwa mapping status (belum_mulai↔Belum Mulai, selesai↔Selesai) bekerja konsisten — bug di sini bisa menyebabkan tugas "hilang" atau status salah.

5.4 Mengapa Integration Testing Perlu Terpisah dari Unit Testing

Unit testing menguji setiap fungsi secara terisolasi, sedangkan integration testing menguji interaksi antar komponen. Contoh nyata: Task.fromMap() bisa lolos unit test (parsing benar), dan _convertFromApi() juga lolos (konversi benar), namun saat digabungkan, mapping status "selesai" → "Selesai" → "selesai" bisa saja gagal karena typo atau case-sensitivity. Integration testing menangkap bug di antarmuka antar modul yang tidak terlihat saat diuji sendiri-sendiri.

5.5 Potensi Bug yang Lolos Unit Test tapi Tertangkap Integration Test

Pada konversi Schedule, jika endTime null di API model, fungsi _calculateEndTime() akan menghitung otomatis (+1 jam). Unit test memverifikasi kalkulasi ini benar secara terisolasi. Namun integration test IT-02 menemukan bahwa saat alur lengkap dijalankan (API → convertFromApi → View model), nilai endTime yang dihitung harus konsisten dengan yang ditampilkan di UI. Tanpa integration test, inkonsistensi antara kalkulasi di model dan di controller bisa lolos.
